# Releasing

Releases are triggered by a version tag matching `v\d+\.\d+\.\d+`. The
`.github/workflows/release.yml` workflow picks up the tag, runs the fast
suite, builds the distribution, publishes to PyPI via trusted publishing,
and creates a GitHub Release with autogenerated notes.

## 1. Checklist

1. **Pick the next semver.** Major for breaking API changes, minor for new
   features, patch for fixes only.
2. **Update version in two places:**
   - `pyproject.toml` → `project.version`
   - `src/registry_review_mcp/__init__.py` → `__version__`
3. **Update `CHANGELOG.md`** with a dated entry at the top using the
   `Added / Changed / Fixed / Performance` sections.
4. **Bump `PROMPT_VERSION`** in `src/registry_review_mcp/tools/evidence_tools.py`
   if (and only if) any extractor prompt template changed in a way that
   alters output. This forces the on-disk response cache to invalidate.
5. **Re-lock baselines.** Run the regression suite, commit the new
   `tests/evaluation/baselines/*-v<next>-*.json` files, archive the
   previous version under `baselines/archived/`.
6. **Validate the build locally:**
   ```bash
   uv build
   ls dist/   # registry-review-mcp-<ver>.tar.gz + registry_review_mcp-<ver>-py3-none-any.whl
   ```
7. **Commit** with a conventional-commit subject:
   ```
   release: v2.5.0
   ```
8. **Tag + push:**
   ```bash
   git tag v2.5.0
   git push origin main v2.5.0
   ```
   GitHub Actions `release.yml` takes over from here.

## 2. PyPI trusted publishing (recommended)

PyPI trusted publishing uses GitHub Actions OIDC tokens — there is no
long-lived API token stored as a GH secret, which materially reduces
supply-chain risk.

### One-time setup

1. Log in at <https://pypi.org/manage/project/registry-review-mcp/settings/publishing/>.
2. **Add a trusted publisher:**
   - Publisher: GitHub
   - Owner: `gaiaaiagent` (or your fork owner)
   - Repository: `regen-registry-review-mcp`
   - Workflow: `release.yml`
   - Environment: `pypi`
3. **Configure the `pypi` environment** in the GitHub repo:
   - Settings → Environments → New environment → `pypi`
   - Add protection rules: required reviewers (matt, shawn), limit to
     the `main` branch + tag refs.

### Fallback — API token (legacy)

If OIDC setup is blocked for any reason:

1. Generate a scoped token at <https://pypi.org/manage/account/token/>
   (scope: `registry-review-mcp` only).
2. Add as `PYPI_API_TOKEN` secret in the GitHub repo.
3. Edit `.github/workflows/release.yml` to pass
   `password: ${{ secrets.PYPI_API_TOKEN }}` to
   `pypa/gh-action-pypi-publish`.
4. Rotate the token annually.

## 4. Rollback

If a release is bad:

1. **Do NOT delete the tag or yank the PyPI release.** Those create
   "phantom" history that confuses downstream installers.
2. Instead: release the next patch version with the fix (e.g. v2.5.1).
3. Optionally mark the bad release as yanked on PyPI via the web UI
   (this warns new installers while preserving reproducibility for
   existing installs).
